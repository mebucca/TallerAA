¿Cuántos problemas ves?


  1. Ruta absoluta con espacios — falla en cualquier otro computador
  2. Nombre de archivo "USAR ESTE v3" en el código — sin control de versiones

Pecado #3 · Resultados producidos a mano


Dos problemas distintos, igual de graves:

  1. Propenso a errores — copiar un número mal, actualizar solo algunas celdas, una fórmula que referencia la columna equivocada
  2. No escala — si el análisis aplica a 6 regiones, tienes que repetir cada paso 6 veces a mano

Propenso a errores


Reinhart & Rogoff (2010): un error en una fórmula de Excel cambió los resultados de un paper que justificó políticas de austeridad en media Europa. El error existió por años sin que nadie lo detectara.

No escalable


La situación típica: el cliente opera en 6 regiones y quiere un reporte por región. Los datos se actualizan cada mes.


Enfoque manual

1. Filtra la hoja por "Metropolitana"
2. Copia las tablas al Word
3. Actualiza los gráficos a mano
4. Repite para Valparaíso
5. Repite para Biobío
6. Repite para las 3 regiones restantes

→ 6 archivos Word distintos
→ Números copiados a mano
→ El mes siguiente: desde cero
→ Garantía: habrá errores y será lento

Enfoque con código

regiones <- unique(clientes$region)

for (r in regiones) {
  quarto_render(
    "reporte.qmd",
    execute_params = list(region = r),
    output_file = paste0(r, ".html")
  )
}
# → 6 reportes idénticos en estructura
# → Nuevos datos, nuevos reportes en un clic
# → Cero errores manuales

Parte II · Principios de un workflow efectivo

¿Qué es el workflow?


Un workflow es el flujo de producción de un análisis: el camino que va desde los datos crudos hasta el reporte final.

El objetivo


Si mi workflow es efectivo, puedo replicar todos mis resultados —figuras, tablas, informes— con un solo clic o una línea de código.


Esto requiere:

  1. Estructura predecible
  2. Datos crudos intocables
  3. Outputs siempre por código

🩺 Problema que resuelve: “¿cuál archivo era el bueno?”, “¿dónde está el dato?”, “¿qué hace este script?”

Buenas prácticas de coding

Práctica #1 · El masterfile

🩺 Problema que resuelve: “¿en qué orden corro los scripts?”, “¿cuál era el punto de entrada?”


Un masterfile (1_masterfile.R) es el único archivo que se corre directamente. Llama a todos los demás en orden:

# 1. Limpiar el entorno
cat("\014"); rm(list = ls())

# 2. Cargar packages
library(pacman)
p_load(tidyverse, knitr, stargazer)

# 3. Definir directorios (solo aquí, una vez)
folder     <- "/Users/bucca/proyecto_nps/"
dircode    <- paste0(folder, "code/")
dirdata    <- paste0(folder, "data/")
dirresults <- paste0(folder, "results/")

# 4. Correr el pipeline en orden
source(paste0(dircode, "2_exploration.R"))
source(paste0(dircode, "3_recoding.R"))
source(paste0(dircode, "4_analyses.R"))

Práctica #2 · Scripts numerados, responsabilidad única

🩺 Problema que resuelve: “¿dónde estaba la limpieza?”, “¿qué hace este script?”


code/
├── 1_masterfile.R      ← llama a todos los demás
├── 2_exploration.R     ← solo exploración
├── 3_recoding.R        ← solo recodificación
├── 4_analyses.R        ← solo análisis y outputs globales
└── 5_analyses_bysucursal.R ← análisis por subgrupo

Cada script hace una sola cosa. Los números indican el orden de ejecución.

  • Si algo falla en el análisis, sabes que el problema está en 4_analyses.R
  • Si quieres cambiar una recodificación, vas directo a 3_recoding.R
  • Nadie tiene que leer 500 líneas para encontrar dónde se filtra algo

Práctica #7 · Comentar el código — estructura del script


Usa encabezados de sección para que el script sea escaneable:

# ══════════════════════════════════════════
# 4_analyses.R
# Análisis NPS por región y sector
# Autor: Mauricio Bucca · mayo 2025
# Inputs:  data/clean/clientes.csv
# Outputs: results/figura_nps.pdf
#          results/tabla_regresion.txt
# ══════════════════════════════════════════


# ── 1. Cargar datos ───────────────────────
clientes <- read_csv(paste0(dirdata, "clientes.csv"))


# ── 2. Descriptivos ───────────────────────
# NPS por región — tabla de resumen
clientes |>
  group_by(region) |>
  summarise(media = mean(nps, na.rm = TRUE)) |>
  arrange(desc(media))


# ── 3. Modelo ─────────────────────────────
# Se usa OLS simple (no mixto) porque no hay
# estructura jerárquica suficiente (n < 30 por grupo)
model1 <- lm(nps ~ satisfaccion + facturacion_mm, data = clientes)

Práctica #8 · Iterar en vez de copiar-pegar

🩺 Problema que resuelve: “el análisis hay que repetirlo para 13 sucursales”

# 1_masterfile.R
sucursales <- unique(clientes$sucursal)  # c("Santiago", "Valparaíso", "Concepción", ...)

for (s in sucursales) {
  cat("=== ANALIZANDO:", s, "===\n")

  # filtra datos de la sucursal s
  clientes_suc <- clientes |> filter(sucursal == s)

  # corre el mismo script — los outputs se guardan con sufijo "_s"
  source(paste0(dircode, "5_analyses_bysucursal.R"))
}


Si se agrega una sucursal, el código lo maneja solo.

  • Guarda todo el historial del proyecto, no solo la versión actual

¿Cómo se usa?


  1. Creas un repositorio en github.com — como crear una carpeta nueva
  2. Lo conectas a tu computador desde Positron — un clic
  3. Trabajas normalmente — editas scripts, corres código, guardas archivos
  4. Cuando algo funciona, haces un commit — una foto del proyecto con un mensaje tuyo
  5. Haces pull — baja los cambios que hiciste en otro computador o que hizo tu colega
  6. Haces push — subes esa foto a GitHub


En Positron todo esto se hace con botones, no con terminal.

Principio DRY — Don’t Repeat Yourself

Si copias y pegas código más de dos veces, hay una mejor manera.


Menos código → menos errores → más confianza.

El problema: repetición manual


media_metro  <- mean(clientes$nps[clientes$region == "Metropolitana"], na.rm = TRUE)
media_valpo  <- mean(clientes$nps[clientes$region == "Valparaíso"],    na.rm = TRUE)
media_biobio <- mean(clientes$nps[clientes$region == "Biobío"],        na.rm = TRUE)


¿Qué pasa cuando hay 6 regiones? ¿O 16? ¿O cuando se agrega otra variable?

La solución: el for loop


for (r in unique(clientes$region)) {
  media <- mean(clientes$nps[clientes$region == r], na.rm = TRUE)
  cat(r, "→ NPS medio:", round(media, 1), "\n")
}


  • eval: false — mostrar el código pero no ejecutarlo
  • warning: false / message: false — silenciar advertencias
  • fig-width / fig-height — tamaño del gráfico
  • cache: true — guardar resultado (útil para chunks lentos)

Valores inline: el número vive en el código


La muestra contiene `{r} nrow(clientes)` empresas 
de `{r} n_distinct(clientes$region)` regiones.

El NPS promedio es **`{r} round(mean(clientes$nps, na.rm=TRUE), 1)`**.


Cambian los datos → los números del texto se actualizan solos.

Nunca más copiar resultados a mano al Word.

¿Cómo se ve en la práctica?


La feature que cambia todo: parámetros


🩺 Problema: “6 regiones × un reporte por cada una = 6 reportes idénticos en estructura, distintos en datos”


❌ A mano

  1. Copiar .qmd 6 veces
  2. Cambiar el filtro en cada uno
  3. Renderizar uno por uno
  4. Resultado: 6 archivos desincronizados
  5. Garantía: habrá un error

✅ Con parámetros

  1. Un reporte maestro
  2. Un loop
  3. Un clic
  4. Resultado: 6 reportes idénticos en estructura
  5. Garantía: cero errores manuales

Quarto con parámetros: cómo se declaran


En el YAML del documento:

---
title: "Estudio Regional"
params:
  region: "Metropolitana"     ← parámetro configurable
  año: 2025
  umbral_promotor: 50
---


Los parámetros se vuelven variables dentro del documento, accesibles como params$region, params$año, etc.

Quarto con parámetros: cómo se usan


## Región: `{r} params$region`

```{r}
datos <- clientes |> 
  filter(region == params$region, year == params$año)

nps_promedio <- mean(datos$nps, na.rm = TRUE)
```

El NPS promedio en **`{r} params$region`** es **`{r} round(nps_promedio, 1)`**.

El umbral para clasificar Promotores es `{r} params$umbral_promotor`.


El mismo documento produce un reporte distinto para cada combinación de parámetros.

Generar todos los reportes con un loop


library(quarto)

regiones <- unique(clientes$region)

for (r in regiones) {
  quarto_render(
    input          = "docs/reporte_regional.qmd",
    execute_params = list(region = r, año = 2025),
    output_file    = paste0("reporte_", tolower(r), ".html")
  )
}

IA en el workflow

Asistencia dentro del pipeline, no fuera de él

¿Qué problema resuelve?


🩺 Problema: “me quedé pegado con un bug y voy a perder 2 horas googleando”, “necesito hacer que el código corra pero no tengo tiempo”


Asistentes como Claude Code, GitHub Copilot o Cursor se integran al workflow.


No reemplazan el pipeline — operan dentro de él.

  • Leen tu código y entienden el contexto
  • Sugieren correcciones, refactorizaciones, tests
  • Ejecutan tareas largas (debuggear, generar 6 reportes, etc.)
  • Documentan funciones, escriben READMEs

Dos formas de integrar IA


Asistente en el IDE

  • Chat dentro de Positron / VS Code
  • Accede al contexto del archivo abierto
  • Ideal para debugging interactivo
  • Sugerencias mientras escribís

Claude Code (terminal)

  • Corre desde la terminal del IDE
  • Lee y escribe archivos
  • Puede ejecutar código
  • Ideal para tareas más largas

El principio que no cambia


La IA opera dentro del pipeline. No lo define. No lo reemplaza.

Un buen pipeline hace que la IA:

  • Funcione mejor — el contexto está claro (estructura, README)
  • Sea auditable — se ve exactamente qué hizo (GitHub)
  • No genere caos — los cambios se rastrean


La IA es una herramienta más en tu toolkit, no el toolkit completo.

# code/01_clean.R
library(tidyverse)

raw <- read_csv("data/raw/encuesta_mayo_2025.csv")

clean <- raw |>
  filter(!is.na(nps)) |>
  mutate(
    nps_grupo = case_when(
      nps < 0  ~ "Detractor",
      nps < 50 ~ "Pasivo",
      TRUE     ~ "Promotor"
    )
  )

write_csv(clean, "data/clean/clientes.csv")
